Lock-free **single-producer / single-consumer** циклическая очередь на C++17.
Статическое хранилище фиксированного размера, без примитивов синхронизации -
только атомарные индексы, - и ручное управление временем жизни объектов.

Очередь рассчитана на конкретный сценарий: **один поток пишет, один читает**
(SPSC). Это её главное ограничение и одновременно то, что позволяет обойтись без
блокировок - вся синхронизация сводится к паре атомарных счётчиков. Типовое
применение во встраиваемых системах - передача данных из обработчика прерывания
в основной цикл или задачу RTOS, либо обмен между ядрами на многоядерном МК.

## Проверка под санитайзерами

Корректность синхронизации проверяется прогоном конкурентного стресс-теста под
**ThreadSanitizer**. TSan моделирует happens-before по модели памяти C++ - через
сами атомарные операции, а не через гарантии конкретного железа. Поэтому
пропущенный `acquire`/`release` он ловит как гонку даже на сильно-упорядоченном
x86, где такой баг в рантайме мог бы и не выстрелить. То есть x86-раннер в CI
полноценно валидирует синхронизацию, релевантную и для слабых моделей памяти
(ARM).

```bash
# ThreadSanitizer
cmake -S . -B build-tsan -DCMAKE_CXX_FLAGS="-fsanitize=thread -g -O1"
cmake --build build-tsan --target ring_queue_tests -j
./build-tsan/ring_queue_tests                 # вся сюита
./build-tsan/ring_queue_tests "[stage_6]"     # только конкурентный стресс-тест
```

```bash
# AddressSanitizer + UBSan (память, утечки, UB)
cmake -S . -B build-asan \
  -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined -fno-sanitize-recover=all -g -O1"
cmake --build build-asan --target ring_queue_tests -j
./build-asan/ring_queue_tests
```

> [!NOTE]  
> Все три прогона (обычный / TSan / ASan+UBSan) гоняются в CI на каждый push -
> бейджи вверху зелёные ровно потому, что workflow реально прошёл, а не потому
> что так написано в README.

## Разбор ТЗ: инженерные решения

#### 1. «безопасную для … двух потоков (один читает, другой пишет)»

=> Это **SPSC** (single-producer / single-consumer).

Конкуренции за общие данные нет, модель синхронизации простая: **продюсер
владеет `m_tail`, консьюмер владеет `m_head`**. Каждый поток пишет только свой
счётчик и лишь читает чужой - <ins>гонки за один и тот же счётчик нет _by
design_</ins>. Несколько писателей или несколько читателей этот класс не
поддерживает - это жёсткое предусловие.

#### 2. «не использующую механизмы взаимной блокировки потоков»

=> Никаких примитивов (mutex, spinlock, futex и т.п.).  
Синхронизация - только атомарные операции с порядком **release / acquire**:

- продюсер публикует данные, затем `m_tail` (release);
- консьюмер читает `m_tail` (acquire), затем данные.

(Полная картина с обоими каналами — в разделе
[Memory ordering](#память-и-порядок-доступа-memory-ordering).)

#### 3. «Память … аллоцируется статически внутри очереди»

=> Член класса `alignas(T) std::byte m_buffer[CAPACITY * sizeof(T)]`.

Никакой кучи, никакого `new[]`. Выравнивание - `alignas(T)`, не зашитые 8/16
байт: берётся реальное требование типа (`alignof(T)`), иначе over-aligned типы
(например, SIMD-векторы) получили бы UB при `placement new`.

`std::byte` (а не `char[]`) - чтобы хранилище честно выражало «сырые байты», без
лишних предположений об арифметике символов.

#### 4. «Два шаблонных параметра: `T`, `CAPACITY`»

=> `template <typename T, std::size_t CAPACITY>`  
, где `CAPACITY` - non-type template parameter.

`static_assert(CAPACITY > 0)`. Хранится ровно `CAPACITY` элементов

> (см. решение про монотонные индексы — без классической «жертвы одного слота»).

#### 5. «от `T` нужны только конструктор копирования и оператор присваивания»

=> Поэтому **нельзя `T m_buffer[CAPACITY]`** - это потребовало бы
default-конструктор, которого у `T` может не быть. Ну а если бы и был, то
насоздавало кучу сырых объектов типа T в `m_buffer`, что противоречит тз -
создавать надо только на push.

Отсюда:

- сырое хранилище + **placement new** на запись;
- **copy-assignment** на извлечение (`out = stored`);
- API `bool push(const T&)` / `bool pop(T& out)` - через out-параметр и `bool`,
  а не `T pop()` / `std::optional<T>` / исключения. Это не требует от `T` ни
  default-конструктора, ни move-семантики, и не бросает исключений из самой
  очереди (важно для embedded-сборок с `-fno-exceptions`).

Тестовый тип `Tracked` устроен ровно по контракту ТЗ: copy-ctor + copy-assign и
**ни одного** default-конструктора, что зафиксировано
`static_assert(!std::is_default_constructible_v<Tracked>)`. Если бы реализация
втихую потребовала что-то сверх ТЗ, тесты бы не собрались.

#### 6. «Очередь поддерживает конструктор копирования и оператор присваивания»

=> [Rule of three](https://en.cppreference.com/w/cpp/language/rule_of_three):

дефолтные версии скопировали бы сырые байты буфера, не вызвав конструктор
копирования `T` (а для занятых атомиков ещё и сломали бы инварианты). Поэтому
копирование - поэлементное, `placement new` по живым элементам источника
`[head, tail)`; `operator=` сначала уничтожает свои элементы, затем копирует
чужие, с защитой от самоприсваивания.

#### 7. «хранятся до затирания; уничтожаются сразу после извлечения»

=> Ручное управление временем жизни:

- запись: `placement new` по сырому адресу слота;
- извлечение: копия в `out`, затем **сразу** явный `~T()`;
- деструктор очереди добивает все оставшиеся живые объекты `[head, tail)`.

Доступ к созданным через `placement new` объектам идёт через `std::launder` -
иначе обращение по указателю, полученному `reinterpret_cast` из байтового
буфера, формально UB (компилятор вправе считать, что там всё ещё «сырые байты»).

## Память и порядок доступа (memory ordering)

Базовое правило - **каждый счётчик принадлежит одному потоку**: продюсер пишет
`m_tail` и только читает `m_head`; консьюмер пишет `m_head` и только читает
`m_tail`. На этом строятся два независимых канала синхронизации.

**Канал 1 - публикация данных (`m_tail`, release -> acquire).**

Писатель: сначала `placement new` в слот, потом `m_tail.store(release)`.  
Читатель: `m_tail.load(acquire)`, потом чтение слота.

`release` запрещает записи данных «переехать» за инкремент индекса;  
`acquire` запрещает чтению данных уехать вперёд индекса.

Без этой пары на слабоупорядоченном ядре (или после переупорядочивания
компилятором) читатель, увидев новый `tail`, мог бы прочитать слот, в который
байты объекта ещё не дописаны => частично записанный / мусорный объект, UB. Пара
release/acquire создаёт happens-before: всё, что записано до `release`,
гарантированно видно после парного `acquire` (распространение — через протокол
кеш-когерентности, без явного «сброса в RAM»).

**Канал 2 — освобождение слота (`m_head`, release → acquire).** Читатель:
`out = *ptr` + `~T()` (слот свободен), потом `m_head.store(release)`. Писатель:
`m_head.load(acquire)` и только потом переиспользует слот под новый
`placement new`. Эта пара гарантирует, что деструктор предыдущего объекта в
слоте завершён **до** конструирования нового. Без неё на обороте кольца (в
стресс-тесте слоты переиспользуются тысячи раз) был бы data race на одной и той
же памяти. Именно поэтому `acquire` на чтении `head` внутри `push` — несущий, а
не «для красоты»: убери его — и TSan поймает гонку.

**`relaxed`** — там, где межпоточной конкуренции нет: чтение собственного
индекса (писатель — свой `tail`, читатель — свой `head`) и весь copy-ctor /
`operator=`, работающие в одном потоке. Нужна только атомарность переменной, не
порядок.

> [!NOTE]  
> Почему release/acquire, а не дефолтный `seq_cst`: `seq_cst` требует единого
> глобального total order над всеми seq_cst-операциями, и за это платится самым
> тяжёлым барьером — на x86 seq_cst-store тянет `MFENCE`/`xchg`, на ARM —
> дополнительные `DMB` с обеих сторон. SPSC такого не требует: два потока
> синхронизируются строго попарно через свои индексы. Поэтому достаточно
> release/acquire — на x86 это вообще обычные load/store без барьера (модель TSO
> даёт acquire/release «бесплатно»), на ARM — лёгкий `DMB` (или `LDA`/`STL` на
> ARMv8).

**False sharing.** `m_head` и `m_tail` разнесены по разным кеш-линиям через
`alignas(64)`. Иначе запись одним потоком своего индекса инвалидировала бы
кеш-линию у другого потока (который читает чужой индекс из той же линии) →
пинг-понг линии между ядрами и просадка пропускной. См. оговорку про embedded
ниже — на одноядерных МК без кеша эта оптимизация смысла не имеет.

## Замечания под микроконтроллеры (embedded)

Очередь писалась с прицелом на bare-metal / RTOS. Что из этого следует и что я
адаптировал бы под конкретный таргет:

- **Без кучи, детерминированно.** Всё хранилище — член класса; нет
  `new`/`malloc`, нет фрагментации, размер известен на этапе компиляции
  (`sizeof(RingQueue<T, N>)`). Ровно то, что нужно там, где динамическая
  аллокация в рантайме нежелательна (Soft realtime) или запрещена (Hard
  realtime).

- **Lock-free - можно из ISR.** Ни одного блокирующего примитива. Типовой
  сценарий на МК: продюсер - обработчик прерывания (DMA-complete, UART RX,
  таймер), консьюмер - основной цикл или задача FreeRTOS. Мьютекс в ISR брать
  нельзя (блокировка, инверсия приоритетов), а здесь `push`/`pop` неблокирующие
  и без запрета прерываний.

- **Дружит с `-fno-exceptions` / `-fno-rtti`.** `push`/`pop` возвращают `bool`,
  сама очередь исключений не бросает (бросить может только копирование `T`).
  Никакого RTTI. Собирается в типичной embedded-конфигурации.

- **Деление `% CAPACITY`.** Физический слот считается как `idx % CAPACITY`. На
  ядрах без аппаратного делителя (Cortex-M0/M0+, ARMv6-M) это разворачивается в
  программное деление (`__aeabi_uidivmod`) - десятки тактов на каждую операцию.
  Можно быстрее, поэтому под конкретный таргет я бы ограничил `CAPACITY`
  степенью двойки (`static_assert((CAPACITY & (CAPACITY - 1)) == 0)`) и заменил
  `% CAPACITY` на `& (CAPACITY - 1)` - одна инструкция. Бонус: степень двойки
  делает `2^width % CAPACITY == 0`, а значит убирает единственный по-настоящему
  острый угол монотонных счётчиков - aliasing слотов на самом переполнении
  счётчика. Но оставил как есть, ТЗ этого не требует.

- **`alignas(64)` — это про многоядерность с кешем.** Даёт выигрыш на Cortex-A /
  x86 и многоядерных МК (STM32H7 M7+M4, RP2040 2×Cortex-M0+). На одноядерных
  Cortex-M0–M4 кеша данных нет вообще — там `alignas(64)` только съедает ~128
  байт RAM на padding, и под такой таргет я бы его убрал. Переносимее, чем
  зашитые `64`, — `std::hardware_destructive_interference_size`.

- **Барьеры на одном ядре.** Даже на in-order Cortex-M, где аппаратных
  переупорядочиваний почти нет, `std::atomic` нужен как барьер **для
  компилятора** — иначе оптимизатор переставит запись данных и инкремент
  индекса, и связка ISR<->main сломается так же, как на многоядерной платформе.

> [!IMPORTANT]  
> Проверка на целевой платформе. На хосте корректность синхронизации закрыта
> прогоном под TSan (выше). Следующий шаг под конкретный МК - прогон
> стресс-теста в конфигурации ISR-продюсер / task-консьюмер на двухъядерном
> таргете (или ISR<->main на одноядерном) с проверкой строгого FIFO; для
> тайминга - тоггл GPIO на `push`/`pop` под логическим анализатором.

## Тесты

14 кейсов, прогоняются на каждый build (`run_tests`) и регистрируются в `ctest`
через `catch_discover_tests`. Покрытие:

- **Жизненный цикл** (`Tracked` со счётчиками constructed/copied/assigned/
  destroyed): объект уничтожается ровно при извлечении; `alive() == 0` после
  работы доказывает отсутствие утечек и двойного уничтожения.
- **FIFO-порядок** и observer-методы (`empty`/`full`/`size`).
- **Границы:** `push` в полную и `pop` из пустой не портят данные и не
  перезаписывают.
- **Обмотка индексов:** 1000 циклов push/pop на `CAPACITY = 4`.
- **"Rule of three":** копия независима от оригинала; copy-ctor и `operator=`
  без утечек и двойного уничтожения; замена содержимого при присваивании;
  безопасное self-assignment.
- **Конкурентность (`[stage_6]`):** 200 000 элементов через очередь
  `CAPACITY = 64` между двумя потоками, проверка строгого FIFO и полного
  опустошения - этот тест, как и все остальные, гоняется под ThreadSanitizer.

Тесты размечены по стадиям (`[stage_1]`…`[stage_6]`). Опция `RING_RUN_ALL_TESTS`
и перемещаемый `#if`-guard - удобство для пошаговой TDD-разработки (намечаешь
тестовые кейсы и проверяешь по одному пока не пройдут все тесты); в финале guard
пуст и собирается все тестовые сценарии.
